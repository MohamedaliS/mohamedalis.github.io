11.1 Learning objectives

By the end of this chapter, the student should be able to:

  1. Define probability and explain what a probability value represents.
  2. Distinguish between classical, empirical, and subjective probability.
  3. Apply the rules for complementary and mutually exclusive events.
  4. Calculate conditional probability and interpret it correctly.
  5. Determine whether two events are independent.

11.2 Introduction

Every decision made under uncertainty involves an implicit probability judgement. A manager approving a new product launch is, in effect, assigning a probability to its success. A financial analyst forecasting revenue is placing a probability on a range of outcomes. A supply chain manager deciding how much stock to hold is estimating the probability that demand will exceed a given level.

Probability is the language in which uncertainty is measured. It assigns a number between zero and one to each possible outcome, where zero means the outcome cannot occur and one means it will certainly occur. This chapter establishes the foundations: what probability is, how it is calculated, and how probabilities for related events are combined.

The dataset for this unit contains 200 customer service incidents recorded at a retail chain, with variables covering incident type, region, time of day, day of week, and resolution outcomes.

11.3 What probability is

A probability is a number between 0 and 1 that expresses the likelihood of a specific outcome or event.

The sample space is the set of all possible outcomes of a process or experiment. An event is any subset of the sample space, meaning any collection of outcomes we are interested in.

Three approaches to assigning probabilities are widely used.

Classical probability applies when all outcomes are equally likely. The probability of an event is the number of outcomes in the event divided by the total number of outcomes in the sample space.

Empirical probability uses observed data. The probability of an event is estimated as the proportion of times that event occurred in a large number of trials or observations. This is the most common approach in business analysis.

Subjective probability reflects personal judgement or expert opinion when historical data are unavailable. It is used in situations such as new product launches or one-off projects.

11.3.1 Probability from observed data

Code 
library(dplyr)

type_probs <- df |>
  count(incident_type, name = "count") |>
  mutate(probability = round(count / sum(count), 3)) |>
  arrange(desc(probability))

knitr::kable(type_probs,
  col.names = c("Incident type", "Count", "Estimated probability"),
  caption   = "Table 11.1: Empirical probabilities for incident types") |>
  kableExtra::kable_styling(bootstrap_options = c("striped", "hover"),
                             full_width = FALSE)
Table 11.1: Empirical probabilities for incident types
Incident type Count Estimated probability
Refund request 73 0.365
Product complaint 70 0.350
Pricing dispute 27 0.135
Theft report 16 0.080
Slip and fall 14 0.070

These probabilities are estimates based on 200 observed incidents. With more data, the estimates would become more stable. The probabilities sum to 1 because the categories are exhaustive and mutually exclusive.

11.4 Basic probability rules

11.4.1 The complement rule

The complement of an event A, written \(A^c\) or \(\bar{A}\), is the event that A does not occur. The sum of the probabilities of an event and its complement must equal 1:

\[P(A^c) = 1 - P(A)\]

If the probability that a randomly selected incident is a refund request is 0.38, then the probability that it is not a refund request is \(1 - 0.38 = 0.62\).

11.4.2 Mutually exclusive events

Two events are mutually exclusive if they cannot both occur at the same time. For mutually exclusive events A and B:

\[P(A \text{ or } B) = P(A) + P(B)\]

Incident types in this dataset are mutually exclusive: a single incident cannot simultaneously be a refund request and a product complaint.

Code 
probs <- setNames(type_probs$probability, type_probs$incident_type)

p_refund   <- probs["Refund request"]
p_complaint <- probs["Product complaint"]
p_either   <- p_refund + p_complaint

cat("P(Refund request)   =", p_refund, "\n")
P(Refund request)   = 0.365
Code 
cat("P(Product complaint) =", p_complaint, "\n")
P(Product complaint) = 0.35
Code 
cat("P(Refund OR Complaint) =", round(p_either, 3), "\n")
P(Refund OR Complaint) = 0.715

11.5 Conditional probability

The conditional probability of event A given that event B has already occurred is:

\[P(A \mid B) = \frac{P(A \cap B)}{P(B)}\]

In plain language: out of all the occasions when B occurred, what proportion also involved A?

11.5.1 Conditional probability from a contingency table

Code 
contingency <- df |>
  count(incident_type, resolved_at_counter) |>
  tidyr::pivot_wider(names_from = resolved_at_counter,
                     values_from = n, values_fill = 0) |>
  mutate(Total = Yes + No,
         P_resolved = round(Yes / Total, 3))

knitr::kable(contingency,
  col.names = c("Incident type", "Resolved", "Not resolved",
                "Total", "P(resolved | type)"),
  caption   = "Table 11.2: Resolution rates by incident type") |>
  kableExtra::kable_styling(bootstrap_options = c("striped", "hover"),
                             full_width = FALSE)
Table 11.2: Resolution rates by incident type
Incident type Resolved Not resolved Total P(resolved &#124; type)
Pricing dispute 8 19 27 0.704
Product complaint 8 62 70 0.886
Refund request 19 54 73 0.740
Slip and fall 5 9 14 0.643
Theft report 4 12 16 0.750

The final column shows \(P(\text{resolved} \mid \text{type})\), the conditional probability that an incident is resolved at the counter given its type. This differs from the overall resolution rate and reveals which incident types are hardest to handle at the first point of contact.

11.6 Independence

Two events A and B are independent if the occurrence of one does not change the probability of the other:

\[P(A \mid B) = P(A)\]

Equivalently, A and B are independent if:

\[P(A \cap B) = P(A) \times P(B)\]

11.6.1 Testing for independence

Code 
# Is resolution independent of whether the incident occurred on a weekend?
df <- df |>
  mutate(weekend = ifelse(day_of_week %in% c("Saturday","Sunday"),
                          "Weekend", "Weekday"))

overall_res <- mean(df$resolved_at_counter == "Yes")

res_by_day <- df |>
  group_by(weekend) |>
  summarise(
    n          = n(),
    p_resolved = round(mean(resolved_at_counter == "Yes"), 3)
  )

knitr::kable(res_by_day,
  col.names = c("Day type", "n", "P(resolved)"),
  caption   = "Table 11.3: Resolution rate by day type") |>
  kableExtra::kable_styling(bootstrap_options = c("striped"),
                             full_width = FALSE)
Table 11.3: Resolution rate by day type
Day type n P(resolved)
Weekday 136 0.772
Weekend 64 0.797
Code 
cat("\nOverall P(resolved) =", round(overall_res, 3), "\n")

Overall P(resolved) = 0.78
Code 
cat("If independent, both conditional probabilities should equal",
    round(overall_res, 3), "\n")
If independent, both conditional probabilities should equal 0.78

If the conditional resolution rates for weekdays and weekends are close to the overall rate, resolution and day type are approximately independent. A large difference would suggest the two events are related.

11.7 Worked example

Note

Scenario. A store manager wants to know the probability that a randomly selected incident on a Saturday is a theft report. Use the dataset to calculate this conditional probability.

Code 
saturday_incidents <- df |> filter(day_of_week == "Saturday")

p_saturday  <- nrow(saturday_incidents) / nrow(df)
p_theft_and_sat <- nrow(df |>
  filter(day_of_week == "Saturday", incident_type == "Theft report")) / nrow(df)
p_theft_given_sat <- p_theft_and_sat / p_saturday

results <- data.frame(
  Quantity = c("P(Saturday)", "P(Theft AND Saturday)", "P(Theft | Saturday)"),
  Value    = round(c(p_saturday, p_theft_and_sat, p_theft_given_sat), 3)
)

knitr::kable(results,
  col.names = c("Quantity", "Value"),
  caption   = "Table 11.4: Conditional probability calculation") |>
  kableExtra::kable_styling(bootstrap_options = c("striped"),
                             full_width = FALSE)

Quantity Value
P(Saturday) 0.170
P(Theft AND Saturday) 0.005
P(Theft &#124; Saturday) 0.029

11.8 Summary

Probability is a number between 0 and 1 expressing the likelihood of an event. Empirical probability estimates this from observed data. The complement rule states that the probabilities of an event and its complement sum to 1. Mutually exclusive events cannot occur simultaneously, and their probabilities add directly. Conditional probability measures the likelihood of one event given that another has occurred. Two events are independent if knowing one occurred provides no information about the other.

## Independence

Two events A and B are **independent** if the occurrence of one does not change the probability of the other:

$$P(A \mid B) = P(A)$$

Equivalently, A and B are independent if:

$$P(A \cap B) = P(A) \times P(B)$$

### Testing for independence

```{r}
#| label: ch11-independence
#| echo: true

df <- df |>
  mutate(weekend = ifelse(day_of_week %in% c("Saturday","Sunday"),
                          "Weekend", "Weekday"))

overall_res <- mean(df$resolved_at_counter == "Yes")
```
